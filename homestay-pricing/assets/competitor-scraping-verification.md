# 竞品数据采集 — 实测验证与技术方案

> 基于browser-agent实测携程/美团等OTA平台后的真实结论

---

## 一、实测结果汇总

### 各平台数据可获取性

| 平台 | 民宿名称 | 评分 | 位置 | 设施 | 价格 | 反爬 | 结论 |
|------|---------|------|------|------|------|------|------|
| 携程 | ✅ | ✅ | ✅ | ✅ | ❌(需登录) | 登录墙 | 登录后可获取 |
| 美团 | ✅ | ✅ | ⚠️ | ⚠️ | ❌(需登录) | 登录墙 | 登录后可获取 |
| 飞猪 | ⚠️ | ⚠️ | ⚠️ | ❌ | ❌(需登录) | 登录墙 | 登录后可获取 |
| 去哪儿 | ❌ | ❌ | ❌ | ❌ | ❌ | 连接超时 | 不推荐 |

### 核心发现

1. **所有主流OTA平台均设登录墙**：未登录状态下只显示"登录看低价"，HTML中无任何价格数据
2. **登录后价格可获取**：登录后触发`getHotelRoomListInland`等API获取价格并动态渲染
3. **基础信息无需登录**：名称、评分、评论数、位置、排名、设施等公开可见
4. **无验证码/IP封锁**：browser-agent正常访问，未被拦截
5. **API签名验证存在**：请求包含`_fxpcqlniredt`和`x-traceID`等签名参数，直接调API不可行

## 二、技术方案设计

### 方案：Browser Agent + 用户授权登录

```
用户授权登录（一次性）
        ↓
Browser Agent自动采集（可重复执行）
        ↓
数据结构化入库
        ↓
多因子定价分析
        ↓
看板展示 + 调价建议
```

### 采集流程详细设计

#### 步骤1：用户授权（一次性）

用户在Qoder中通过browser-agent打开OTA网站并手动登录：
- 携程：m.ctrip.com 或 www.ctrip.com
- 美团：i.meituan.com 或 www.meituan.com
- 飞猪：www.fliggy.com

登录后session保存在browser中，后续采集自动使用。

#### 步骤2：配置竞品列表（一次性）

用户在pricing-config中填写竞品名称和平台链接，例如：
```
竞品1: 云栖山居 — 携程链接
竞品2: 溪谷民宿 — 美团链接
竞品3: 湖畔小筑 — 飞猪链接
```

#### 步骤3：自动采集（按需/定时）

Browser Agent执行以下流程：
1. 打开竞品1的携程详情页
2. 等待页面加载完成（价格渲染）
3. 提取房型、价格、促销、评分数据
4. 切换到竞品2的美团详情页
5. 重复提取
6. 汇总输出结构化数据

#### 步骤4：数据入库与分析

采集数据进入定价模型，生成调价建议并更新看板。

## 三、Browser Agent 采集指令设计

### 采集单个竞品价格的browser流程

```
1. navigate(竞品详情页URL)
2. wait_for_selector(".room-price-item" 或 ".hotel-price")  // 等待价格渲染
3. extract_data({
     房型: ".room-name",
     价格: ".room-price .price-num",
     促销: ".promo-tag",
     评分: ".hotel-score"
   })
4. 返回结构化JSON
```

### 批量采集流程

```
对于每个竞品:
  对于每个平台:
    navigate(竞品在该平台的URL)
    wait_for_load()
    extract_pricing_data()
    汇总到结果集
返回完整竞品价格表
```

## 四、数据可靠性保障

### 数据来源可信度

| 数据项 | 来源 | 可信度 | 说明 |
|--------|------|--------|------|
| 竞品名称 | OTA平台页面 | 高 | 直接从页面提取 |
| 竞品价格 | OTA平台页面(登录后) | 高 | 与APP端显示一致 |
| 竞品评分 | OTA平台页面 | 高 | 公开数据 |
| 促销活动 | OTA平台页面(登录后) | 高 | 登录后可见 |
| 入住率预测 | 历史数据推算 | 中 | 基于自有订单数据 |
| 距入住天数 | 系统计算 | 高 | 确定性计算 |

### 数据时效性

| 采集频率 | 适用场景 | 说明 |
|---------|---------|------|
| 每日1次 | 日常工作日 | 获取当日竞品价格快照 |
| 每日2-3次 | 节假日/旺季 | 价格波动大，需更频繁监控 |
| 按需触发 | 突发情况 | 如发现订单异常波动时 |

### 数据一致性校验

- 同一竞品在多个平台的价格交叉对比
- 价格突变告警（24小时内涨跌超20%标记为异常需人工确认）
- 采集失败自动重试（最多3次）

## 五、风险与应对

| 风险 | 等级 | 应对措施 |
|------|------|---------|
| 登录态过期 | 中 | 检测到价格不显示时提示用户重新登录 |
| OTA改版导致选择器失效 | 中 | 设计多套备选选择器，失败时通知更新 |
| 采集频率过高被限 | 低 | 控制采集间隔>30秒，每日不超过10次 |
| 价格与APP端不一致 | 低 | 标注数据来源为"Web端采集"，可能与APP端有差异 |
| 账号安全风险 | 中 | 建议用户使用专用采集账号，勿用个人主账号 |

## 六、与原方案的对比

| 维度 | 原方案(WebSearch/WebFetch) | 新方案(Browser Agent) |
|------|---------------------------|----------------------|
| 价格数据 | ❌ 拿不到 | ✅ 登录后可获取 |
| 数据可信度 | 低(搜索引擎摘要) | 高(页面直接提取) |
| 实时性 | 低(搜索结果有延迟) | 高(实时页面数据) |
| 实现复杂度 | 低 | 中(需browser-agent) |
| 用户操作 | 无需操作 | 需一次性登录授权 |
| 可持续性 | 不可靠 | 可靠(只要登录态有效) |
