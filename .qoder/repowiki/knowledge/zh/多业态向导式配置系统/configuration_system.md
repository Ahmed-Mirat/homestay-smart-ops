### 1. 核心架构与模式
该仓库采用**“向导式初始化 + 集中化 JSON 存储 + 编程式写入”**的配置管理模式。系统不依赖传统的 `.env` 或 `yaml` 文件，而是通过一个交互式的安装向导（Setup Wizard）引导用户输入业务数据，最终将配置持久化为结构化的 JSON 文件。

- **多业态适配**：通过 `propertyType` 字段区分民宿、公寓、酒店、中医馆等不同业态，配置结构动态分支。
- **状态驱动**：使用 `setup-state.json` 跟踪安装进度，支持断点续传和分步修正。
- **环境自检**：提供 `check-env.js` 脚本，在运行时对 Node 版本、依赖包、配置文件完整性及网络连通性进行 10 项自动化检查。

### 2. 关键配置文件
- **`_shared/config.json`**：**核心业务配置库**。存储商户名称、地址、房型/诊疗项目详情、价格体系、会员规则等所有业务逻辑所需的静态数据。
- **`_shared/setup/setup-state.json`**：**安装状态机**。记录当前安装步骤、完成状态、最后修改时间以及当前的 `propertyType`。
- **`_shared/data/staff.json`**：**人员信息库**。独立存储员工、医师、管家等联系人信息，支持按角色分类管理。
- **`_shared/homestay-suite.json`**：**套件元数据**。定义当前套件的名称、版本以及支持的商户类型白名单（`setupTypes`）。

### 3. 配置加载与管理逻辑
- **配置写入器 (`config-writer.js`)**：
  - 这是配置系统的“唯一入口”。严禁商户直接编辑 JSON 文件。
  - 提供了一套语义化的 API（如 `addTreatment`, `updatePricing`, `addMembershipTier`），负责执行“读取→合并→校验→写入”的原子操作。
  - 内置严格的数据校验逻辑（如手机号格式、金额正数校验、必填项检查），确保配置数据的合法性。
- **安装向导 (`SETUP-WIZARD.md`)**：
  - 定义了 5 步标准化流程：环境就绪 → 基础信息 → 业务详情 → 规则标准 → 团队与通知。
  - 问卷定义位于 `_shared/setup/questions/` 目录下，按业态（如 `tcm-clinic`）和通用模块（`_common`）组织。
- **知识库生成 (`kb-generator.js`)**：
  - 在配置完成后，自动根据 `config.json` 中的业务数据生成 Markdown 格式的知识库，供智能客服调用。

### 4. 开发者规范
- **禁止手动修改 JSON**：所有配置变更必须通过 `config-writer.js` 暴露的方法进行，以触发状态更新和知识库重建。
- **业态隔离**：新增业态时，需在 `homestay-suite.json` 中注册，并在 `questions/` 下创建对应的问卷目录，同时在 `config-writer.js` 中补充专用的写入方法。
- **校验前置**：在 `config-writer.js` 中增加新字段时，必须同步更新 `validators` 模块，确保脏数据无法进入配置库。
- **状态同步**：任何涉及业务结构的变更（如增删房型/项目），都应调用 `touchSetupState()` 更新最后修改时间，以便外部系统感知配置变动。
